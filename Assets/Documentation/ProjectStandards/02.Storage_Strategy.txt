## UPDATE-RHYTHMUS

### Täglich (während aktiver Entwicklung)
```
1. Arbeitsprotokoll aktualisieren
   → Aufgaben markieren (geplant/in Bearbeitung/erledigt)
   → Screenshot machen

2. Git Commit
   → Mindestens 1 sinnvoller Commit pro Tag

3. STATE.md aktualisieren (bei Session-Ende)
   → Aktueller Stand
   → Nächste Aktion
   → Git Status
```

### Bei Meilensteinen (Phase-Ende)
```
1. Projektplan aktualisieren
   → Phase als Done markieren
   → Nächste Phase vorbereiten

2. STATE.md komplett aktualisieren
   → Fortschritt-Checkboxen
   → Session History

3. GDD v2 (nach Phase 2)
   → Design-Änderungen dokumentieren
```

### Bei neuen Entscheidungen
```
1. Architecture_Decisions.txt erweitern
   → Neues ADR hinzufügen
   → In Log-Tabelle eintragen
```

## FÜR NEUE SESSION (Quick Reference)

**Lies in dieser Reihenfolge:**
1. `STATE.md` → Wo sind wir?
2. `CLAUDE.md` → Was ist das Projekt?
3. Letzte Commit-Message → Was war der letzte Schritt?

**Das reicht für 90% der Fälle.**

Bei komplexeren Fragen:
- `Architecture_Decisions.txt` → Warum-Entscheidungen
- `GDD` → Game Design Details
- `Projektplan` → Was kommt als Nächstes

---

## ANTI-PATTERNS (Was wir NICHT machen)

❌ Kein komplexes Tier-System
❌ Keine wöchentlichen/monatlichen Audits
❌ Keine Master-Index-Datei
❌ Keine Archive-Ordner für alte Dokumente
❌ Keine redundanten Informationen

**Warum?** Solo-Projekt, 4 Wochen Zeitraum, kein Overhead nötig.
